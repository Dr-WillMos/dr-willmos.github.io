视盾项目扩展

当前技术栈

层级 技术栈 关键组件
客户端 Android (API 29+) MediaProjection, MediaRecorder, ForegroundService, WindowManager
后端（现有） Spring Boot 3.5.14, Java 25 MyBatis, MySQL, Redis (Lettuce), RestClient
AI 服务（其他人） Python 3.10+, FastAPI CLIP (ViT-B/32), Whisper (base), FFmpeg

零级优先

视频上传安全与校验

  • 目标：防止无效/恶意视频消耗 AI 资源
  • 技术栈：Spring Boot (MultipartFile 校验), Java 25
  • 实现方案：
    • 增加 @VideoValidation 自定义校验器，检查：
      • 文件扩展名 .mp4
      • MIME 类型 video/mp4
      • 文件大小 ≤ 10MB（已有）
      • 视频时长 ≤ 5 秒（通过 FFmpeg 或 Metadata 解析）
    • 校验失败返回 400 及明确错误码
    • 可能需要开发用户登录。

基础鉴权（API Key）

  • 目标：防止未授权客户端调用后端接口，攻击者可能拆包或者抓包拿到api后攻击
  • 技术栈：Spring Security
  • 实现方案：
    • 为每个客户端颁发 X-API-Key
    • 后端使用拦截器或 OncePerRequestFilter 校验
    • 配置化密钥（环境变量）

视频编码压缩（客户端）

  • 目标：降低上传流量，提升低端机兼容性

  • 技术栈：Android MediaRecorder

  • 实现方案：

    • 修改 prepareRecorder()：
    • 保持 H.264 + MP4 封装

可观测性基础（日志 + 健康检查）

  • 目标：便于排查问题，演示时展示系统状态
  • 技术栈：Spring Boot Actuator, Micrometer,
  • 实现方案：
    • 添加依赖 spring-boot-starter-actuator
    • 暴露 health, info, metrics 端点（可选鉴权）
    • 在 DetectionService 关键节点打印日志（含耗时）
    • 自定义 AIHealthIndicator 检测 Python 服务连通性

Docker Compose 一键启动

  • 目标：评审/演示环境快速搭建，且能够开箱即用快速部署。
  • 技术栈：Docker, Docker Compose
  • 实现方案：
    • 编写 Dockerfile 用于 Spring Boot 后端
    • 编写 docker-compose.yml 集成：
      • MySQL 8.0
      • Redis 7-alpine
      • Spring Boot 后端

一级优先

历史检测记录客户端

我想过，本地缓存是必要的，在每次启动的时候就将记录(无非就是文本也没多大)存在本地，可以加快速度也减轻mysql压力

  • 目标：用户可追溯过往监测结果
  • 技术栈：Android (RecyclerView, Room), Retrofit
  • 实现方案：
    • 后端新增接口 GET /api/v1/history?deviceId=&page=&size=
    • 返回分页检测记录（id, riskLevel, score, createdAt）
    • 客户端使用 Room 本地缓存 + 下拉刷新
    • UI 按风险等级颜色区分

RocketMQ 异步持久化

  • 目标：降低检测接口响应延迟，解耦 AI 调用与 DB 写入
  • 技术栈：RocketMQ, Spring Boot RocketMQ Starter
  • 实现方案：
    • 启用 MQ（配置 rocketmq.name-server）
    • 检测成功后，发送消息到 detection-record topic
    • 独立 @RocketMQMessageListener 消费者执行 insert

多层次名单（灰度/临时黑名单）

  • 目标：如果黑名单来自权威发布者，如抖音官方，那我们是否可以用“Blacklist命中”一票否决而不用ai调用？这确实能够减少无效 AI 调用
  • 技术栈：Redis (Set / Sorted Set), Spring Data Redis
  • 实现方案：
    • 全局黑名单（现有 blacklist:authors）：永久，人工维护
    • 临时黑名单：blacklist:temp:authors（Sorted Set, score=过期时间戳），自动清理

流式上传（分段上传）

  • 目标：还是需要并发，这才能提高效率减少单次上传等待，提供更快的部分反馈。
  • 技术栈：Android (OkHttp multipart), Spring Boot
  • 实现方案：
    • 客户端：5 秒录屏切成 5 个 1 秒片段，依次上传
    • 后端：新增 /api/v1/detect/stream 接口，接收 fragmentIndex 和 totalFragments
    • Redis 暂存片段，最后一个片段到达后组装并调用 AI
    • 可选：每个片段单独 AI 分析

5. 基础可视化后台（管理端）

  • 目标：管理员需要有一个简单的后台操作，来查看服务器压力，甚至是导出数据交付给警方做反诈工作等
  • 技术栈：Spring Boot +Vue 3
  • 实现方案（简化版）：
    • 使用 Vue快速搭建 3 个页面：
      • 检测记录列表（分页、筛选）
      • 黑名单管理（增删、批量导入）
      • 系统状态（AI 连通性、Redis 命中率）
    • 复用现有 Controller，增加 @Controller 视图方法
    • 基本认证（HTTP Basic）

二级优先

语音播报 + 紧急联系人通知

  • 目标：高危场景下主动触达用户亲友
  • 技术栈：Android TextToSpeech, 短信/推送 API
  • 实现方案：
    • 高危判定后自动语音朗读警告用户
